Официальное руководство по системе CppMutTest

1. Название и краткое описание

CppMutTest — это кроссплатформенная автоматизированная система модульного тестирования для приложений на C++, основанная на мутационном анализе. Она предназначена для оценки качества тестового набора путем внесения изменений (мутаций) в исходный код и проверки способности тестов выявлять эти изменения. Система помогает разработчикам находить слабые места в тестах и повышать надежность программного обеспечения.

2. Поддерживаемые фреймворки тестирования

CppMutTest поддерживает два популярных фреймворка для модульного тестирования:

  • GoogleTest (gtest): широко используемый фреймворк от Google для написания и выполнения тестов.
  • Boost.Test: библиотека из набора Boost для создания модульных тестов.

Все необходимые файлы фреймворков встроены в систему и доступны сразу.

3. Поддерживаемые компиляторы

Система совместима со следующими компиляторами:

  • g++: компилятор из набора GCC (GNU Compiler Collection).
  • clang++: компилятор из проекта LLVM, обеспечивающий высокую производительность и точную диагностику.

При использовании OS Windows необходимо прописать пути до компиляторов в PATH.

4. Установка и запуск

Установка

  1. Скачать архив CppMutTest из репозитория разработчика.
  2. Распаковать архив в указанную директорию.

Запуск

  1. Запустите исполняемый файл CppMutTest.
  2. После запуска откроется окно предварительных настроек и логов приложения, где вы сможете выбрать проект для анализа.

Дополнительно

  1. Убедитесь, что набор файлов приложения полный.
  2. Убедитесь, что в вашей системе установлен хотя бы один из поддерживаемых компиляторов. При использовании OS Windows, укажите пути к ним в PATH.

5. Использование системы

5.1 Выбор проекта

После запуска программы, в меню «Предварительные настройки и логи» вставьте абсолютный путь до папки с вашим проектом в поле «Каталог с проектом» или выберите папку проекта через системный выбор после нажатия на кнопку «…» рядом с полем.

5.2 Предварительная настройка для мутационного анализа

  1. После выбора проекта система проанализирует его структуру (см. раздел «Общие правила для структуры проекта пользователя») и оповестит о возможности дальнейшей работы.
  2. В меню «Предварительные настройки и логи» → «Фреймворк тестирования» выберите используемый фреймворк тестирования (см. раздел «Правила для модульных тестов пользователя» и «Ограничения для модульных тестов пользователя»).
  3. В меню «Предварительные настройки и логи» → «Компилятор» выберите компилятор. Выбор компилятора можно менять в процессе работы программы.
  4. В меню «Предварительные настройки и логи» нажмите «анализ >>». Система проведет анализ исходных файлов и определит список классов пользователя, доступных для анализа.
  5. Если система определила список доступных для анализа классов, то в открывшемся меню «Анализ» выберите класс, щелкнув по нему в списке. Отобразится его исходный код.
  6. Укажите операторы мутации (например, арифметические, логические и т.д.) с помощью соответствующих флажков в «Типы мутаций».
  7. Установите количество мутаций для генерации с помощью ползунка или поля ввода.
  8. Нажмите «Запустить мутационный анализ».

5.3 Результаты мутационного анализа

После успешного завершения мутационного анализа или предварительной остановки, система отображает список мутантов с цветовой индикацией состояния (красный – мутант «выжил», системный – мутант «убит»). Код каждого мутанта можно посмотреть, кликнув по нему в списке.

При выборе мутанта, помимо отображения его исходного кода и выделения строк с изменениями, отображается результат его тестирования.

Система также формирует Mutation Score Indicator (MSI) – показатель качества тестов в процентах (округлен до одного знака после запятой), а также MSI по каждому применённому типу мутации в анализе.

Помимо просмотра исходного кода оригинального файла и мутанта, вы можете сравнить их, нажав на «Оригинал».

5.4 Советы по использованию

  • Мутационный анализ – ресурсоёмкий процесс. Настоятельно рекомендуется закрыть другие процессы в системе и отключить антивирус. Т.к. система создаёт внешние процессы компиляции, антивирусные системы занимают значительную часть процессорного времени на проверку каждого, что может пагубно сказаться на производительности анализа и его успешном завершении.
  • Анализируйте выживших мутантов, чтобы выявить непокрытые тестами участки кода.
  • Выбирайте операторы мутации, соответствующие логике вашего проекта.
  • Регулярно запускайте анализ для отслеживания качества тестов по мере развития проекта.
  • По возможности, используйте компилятор g++, т.к. его производительность выше.
  • Проверьте, что оригинальные тесты корректно выполняются на исходном коде проекта. Их можно запустить с помощью «Анализ» → «Запуск тестов на SRC»
  • Разработчик рекомендует использовать утроенное количество мутаций относительно строк исходного кода анализируемого класса, но не менее 300, для показательного результата MSI.

6. Правила для организации структуры проекта пользователя и ограничения

6.1 Общие правила для структуры проекта пользователя

  1. Выбранная пользователем корневая папка с проектом должна содержать:
    • Исходные файлы (.cpp, .h) в корневой директории.
    • Папку tests с файлами модульных тестов (.cpp).
  2. Проект должен включать зависимости только стандартных библиотек C++ 17 и старше.
  3. В выбранной пользователем корневой папке недопустимо наличие файла с функцией main!

6.2 Правила для классов пользователя

  1. Классы должны быть объявлены в заголовочных файлах (.h).
  2. Реализация методов должна находиться в файлах .cpp, которые подключают соответствующий заголовочный файл через #include "class.h".
  3. Методы классов в файлах .cpp должны быть записаны в формате ClassName::methodName.
  4. Классы должны быть уникальными.
  5. Недопустимо наличие нескольких .cpp файлов для одного класса.

6.3 Правила для модульных тестов пользователя

Для GoogleTest:

  1. Тесты должны использовать макрос TEST(ClassName, TestName), где ClassName – точное название класса из определения.
  2. Имя класса в тестах должно точно совпадать с именем тестируемого класса.
  3. Возможно наличие нескольких файлов с модульными тестами для одного класса.

Для Boost.Test:

  1. Каждый тестовый файл должен содержать директиву #define BOOST_TEST_MODULE ClassName, где ClassName — имя тестируемого класса.
  2. Тесты определяются с помощью макроса BOOST_AUTO_TEST_CASE(TestName).

6.4 Ограничения для модульных тестов пользователя

  1. Каждый тестовый файл должен соответствовать только одному классу.
  2. Для Boost.Test не допускается наличие нескольких тестовых файлов для одного класса.

7. Поддерживаемые операторы мутации

CppMutTest поддерживает:

  • Мутации арифметических операторов «+», «-», «*», «%», «/», «++», «--», включая операторы составных присваиваний «+=», «-=», «/=», «*=», «%=».
  • Мутации логических операторов «!», «||», «&&» и операторов сравнения.
  • Мутации целочисленных и вещественных десятичных констант.
  • Мутации строковых констант "…", включая мутации сырых строк R”delimiter(…)delimiter”.
  • Мутации обработки и выбрасывания исключений, включая их удаление: удаление блоков catch и выражений throw.

8. Системные требования

Для работы CppMutTest необходимы:

  • Операционная система: Windows, Linux или macOS.
  • Компиляторы: установленные g++ (GCC) или clang++ (LLVM).
  • ОЗУ: минимум 4 ГБ, рекомендуется 8 ГБ.
  • Дисковое пространство: не менее 2 ГБ свободного места на диске для приложения и временных файлов рабочей директории.
  • Процессор: многоядерный процессор для оптимальной производительности параллельного анализа.
